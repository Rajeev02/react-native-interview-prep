### Q1. `type` vs `interface` — declarative differences + merge behavior.
**🎯 One-liner** — Same 90% of the time; `interface` merges, `type` composes.
**🧠 Mental model** — `interface` = a **document you can keep adding pages to**. `type` = a **finished blueprint** — to extend it you make a new one.
**📖 Theory**
- Both describe object shapes; both extend.
- `interface` supports **declaration merging** (multiple declarations combine).
- `type` supports **unions, intersections, mapped, conditional** — all type-algebra.
- `interface` only describes object/function shapes; `type` describes anything.
**💻 Code**
```ts
interface User { id: string }
interface User { name: string }   // ✅ merged → { id; name }

type T = { id: string };
// type T = { name: string };     // ❌ duplicate identifier
type Both = T & { name: string }; // composition
type Status = 'on' | 'off';        // unions only via `type`
```
**⚖️ Tradeoff** — merging is great for libraries, dangerous for app code (silent surface changes).
**💥 Prod failure** — a third-party `.d.ts` augmented the app's `Window` interface and silently widened a critical field to `any`.
**🔁 Memory hook** — **"interface = open file, type = sealed envelope."**
**🔗 See:** [docs/03 §types vs interfaces](../../docs/03-typescript.md)

### Q2. When to prefer `interface` over `type`.
**🎯 One-liner** — Public API surfaces and class contracts.
**🧠 Mental model** — Library consumers should be able to **extend your shapes** without forking; that's exactly what merging gives you.
**📖 Theory**
- Public package types → `interface` (downstream can `declare module 'x' { interface Foo { extra: string } }`).
- Class implements → `interface` reads idiomatically.
- App-level domain models with no extension story → `type` is fine.
- Need union/intersection/conditional → `type` (you have no choice).
**💻 Code**
```ts
// Library
export interface ButtonProps { label: string; }

// Consumer augments
declare module 'my-lib' {
  interface ButtonProps { tracking?: string; }
}
```
**⚖️ Tradeoff** — augmentation is a footgun if the augmenter and library disagree on intent.
**💥 Prod failure** — two micro-frontends both augmented the same library interface; types compiled, runtime exploded.
**🔁 Memory hook** — **"public = interface, private = type."**
**🔗 See:** [docs/03](../../docs/03-typescript.md)

### Q3. Generics — write `identity<T>`.
**🎯 One-liner** — A function that **forwards a type without losing it**.
**🧠 Mental model** — `T` is a **placeholder envelope** — whatever the caller puts in is what they get out.
**📖 Theory**
- `<T>` introduces a type parameter at function/type/class scope.
- TS infers `T` from the call site; you rarely pass it explicitly.
- Without generics, you'd lose the type to `any` or `unknown`.
**💻 Code**
```ts
function identity<T>(x: T): T { return x; }
const n = identity(42);     // T = number
const s = identity('hi');   // T = string
```
**⚖️ Tradeoff** — every extra type param is cognitive load; only add when output type depends on input.
**💥 Prod failure** — `function wrap(x: any): any` swallowed an array → callers got `.map is not a function` only at runtime.
**🔁 Memory hook** — **"in = out, no `any` lost."**
**🔗 See:** [docs/03 §generics](../../docs/03-typescript.md)

### Q4. Generic constraint — `<T extends { id: string }>` example.
**🎯 One-liner** — Restrict the placeholder to shapes you can actually use.
**🧠 Mental model** — A bouncer at the type door: "no entry without an `id`."
**📖 Theory**
- `extends` here is **subtype check**, not inheritance.
- Lets you safely access `T['id']` inside the function.
- Combine with `keyof` for property-picking utilities.
**💻 Code**
```ts
function byId<T extends { id: string }>(items: T[], id: string): T | undefined {
  return items.find(i => i.id === id);
}
byId([{ id: 'a', n: 1 }], 'a'); // returns the object, type is { id; n }
```
**⚖️ Tradeoff** — over-constraining kills reuse; under-constraining requires `as` later.
**💥 Prod failure** — generic helper assumed `.id`, a refactor renamed to `_id`, every call site failed silently in JS interop.
**🔁 Memory hook** — **"extends = must-have, not is-a."**
**🔗 See:** [docs/03](../../docs/03-typescript.md)

### Q5. `keyof T` — what it returns.
**🎯 One-liner** — A **union of T's property names** as string literals.
**🧠 Mental model** — Hand the type `{a; b; c}` and you get `'a' | 'b' | 'c'`.
**📖 Theory**
- Returns the literal union of public keys.
- Combined with indexed access: `T[keyof T]` = union of value types.
- Powers `Pick`, `Omit`, `Record`, mapped types.
**💻 Code**
```ts
type User = { id: string; age: number };
type K = keyof User;        // 'id' | 'age'
type V = User[keyof User];  // string | number

function get<T, K extends keyof T>(o: T, k: K): T[K] { return o[k]; }
get({ id: 'x', age: 9 }, 'age'); // type: number
```
**⚖️ Tradeoff** — index signatures (`{ [k: string]: V }`) make `keyof` widen to `string | number`, which is rarely what you want.
**💥 Prod failure** — `keyof` on a `Record<string, T>` returned `string`, allowing typos through to runtime.
**🔁 Memory hook** — **"keyof gives keys, T[K] gives the value."**
**🔗 See:** [docs/03 §keyof](../../docs/03-typescript.md)

### Q6. `typeof variable` in type position.
**🎯 One-liner** — Capture the **shape of a value** as a type.
**🧠 Mental model** — Photocopy the runtime shape into compile-time.
**📖 Theory**
- `typeof x` in a type expression returns x's inferred type.
- Use to derive types from constants, configs, hooks.
- Combine with `keyof typeof obj` for "literal union of an object's keys."
**💻 Code**
```ts
const config = { env: 'prod', region: 'in' } as const;
type Config = typeof config;          // { readonly env: 'prod'; readonly region: 'in' }
type Key = keyof typeof config;       // 'env' | 'region'
```
**⚖️ Tradeoff** — without `as const`, you get widened types (`string` instead of literals).
**💥 Prod failure** — `typeof THEME` without `as const` widened `'dark' | 'light'` to `string` → switch-exhaustiveness silently broke.
**🔁 Memory hook** — **"value → type via typeof, freeze widening with `as const`."**
**🔗 See:** [docs/03](../../docs/03-typescript.md)

### Q7. `Pick<T,K>` vs `Omit<T,K>` — implement Pick.
**🎯 One-liner** — Pick = keep these keys, Omit = drop these keys.
**🧠 Mental model** — Two filters on a property bag — whitelist vs blacklist.
**📖 Theory**
- `Pick<T, K extends keyof T>` keeps only `K`.
- `Omit<T, K>` removes `K` (uses `Exclude` under the hood).
- Both produce **new types**, originals unchanged.
**💻 Code**
```ts
type MyPick<T, K extends keyof T> = { [P in K]: T[P] };

type User = { id: string; name: string; password: string };
type Public = MyPick<User, 'id' | 'name'>;       // strip secrets
type Safe = Omit<User, 'password'>;
```
**⚖️ Tradeoff** — `Omit` doesn't error on unknown keys (uses `Exclude`); `Pick` does.
**💥 Prod failure** — `Omit<User, 'pasword'>` (typo) silently kept the field — leaked password into API response.
**🔁 Memory hook** — **"Pick = whitelist, Omit = blacklist; Pick yells, Omit shrugs."**
**🔗 See:** [docs/03 §utility-types](../../docs/03-typescript.md)

### Q8. `Partial<T>` vs `Required<T>` — implement Partial.
**🎯 One-liner** — Make every prop optional / required, respectively.
**🧠 Mental model** — A switchboard for `?` modifiers across all keys.
**📖 Theory**
- `Partial<T>` adds `?` to every prop.
- `Required<T>` removes `?` from every prop.
- Mapped type with `?` modifier (`+?` / `-?`).
**💻 Code**
```ts
type MyPartial<T> = { [K in keyof T]?: T[K] };
type MyRequired<T> = { [K in keyof T]-?: T[K] };

function update<T>(state: T, patch: Partial<T>): T { return { ...state, ...patch }; }
```
**⚖️ Tradeoff** — `Partial` lets you `update(s, {})` which may not be desired; consider `AtLeastOne<T>`.
**💥 Prod failure** — `Partial<User>` on a PATCH endpoint allowed empty body → server overwrote with defaults.
**🔁 Memory hook** — **"`?:` in, `-?:` out."**
**🔗 See:** [docs/03](../../docs/03-typescript.md)

### Q9. `Readonly<T>` — implement.
**🎯 One-liner** — Mark every prop `readonly`; assignments fail at compile time.
**🧠 Mental model** — A glass case around the object.
**📖 Theory**
- Mapped type with `readonly` modifier.
- Shallow only — nested objects still mutable.
- For deep, write `DeepReadonly<T>` recursively.
**💻 Code**
```ts
type MyReadonly<T> = { readonly [K in keyof T]: T[K] };

type DeepReadonly<T> = {
  readonly [K in keyof T]: T[K] extends object ? DeepReadonly<T[K]> : T[K];
};
```
**⚖️ Tradeoff** — only enforces in TS; pure JS callers bypass it.
**💥 Prod failure** — Redux state typed `Readonly` but a third-party reducer mutated nested arrays → time-travel debug broken.
**🔁 Memory hook** — **"readonly = locked binding, not locked object."**
**🔗 See:** [docs/03](../../docs/03-typescript.md)

### Q10. `Record<K,V>` — implement.
**🎯 One-liner** — A typed dictionary: every key in `K` maps to `V`.
**🧠 Mental model** — A row of labeled boxes, all the same shape inside.
**📖 Theory**
- `K` is usually a string-literal union or `string`/`number`.
- `V` is the value shape.
- Built from a mapped type.
**💻 Code**
```ts
type MyRecord<K extends keyof any, V> = { [P in K]: V };

type Status = 'active' | 'paused' | 'dead';
const labels: Record<Status, string> = { active: 'A', paused: 'P', dead: 'D' };
// missing a key → compile error
```
**⚖️ Tradeoff** — `Record<string, V>` allows any key — defeats exhaustiveness.
**💥 Prod failure** — `Record<string, Handler>` accepted typo'd event names with no error → handler never fired.
**🔁 Memory hook** — **"Record = same value, many keys."**
**🔗 See:** [docs/03](../../docs/03-typescript.md)

### Q11. `ReturnType<typeof f>` — example.
**🎯 One-liner** — Extract the **return type** of a function from its value.
**🧠 Mental model** — Reverse-engineer what a function gives back without re-stating it.
**📖 Theory**
- `ReturnType<T extends (...a: any) => any>` uses `infer R` internally.
- Pair with `typeof fn` when you only have the value.
- Critical for typing custom hooks / selectors.
**💻 Code**
```ts
function useUser() { return { id: '1', name: 'A' }; }
type User = ReturnType<typeof useUser>; // { id: string; name: string }
```
**⚖️ Tradeoff** — Coupling: refactoring `useUser` silently changes `User` everywhere.
**💥 Prod failure** — Hook return widened from `User` to `User | null` — every consumer lost the null-check warning until runtime.
**🔁 Memory hook** — **"ReturnType = infer the back side."**
**🔗 See:** [docs/03 §infer](../../docs/03-typescript.md)

### Q12. `Parameters<typeof f>` — example.
**🎯 One-liner** — Extract a function's **parameter tuple** as a type.
**🧠 Mental model** — Snapshot the front-door arguments.
**📖 Theory**
- Returns a tuple type (so you can spread it).
- Useful for HOF wrappers, decorators, telemetry.
- Sibling utilities: `ConstructorParameters`, `ThisParameterType`.
**💻 Code**
```ts
function api(url: string, init?: RequestInit) { return fetch(url, init); }
type Args = Parameters<typeof api>; // [string, RequestInit?]

function withLog<F extends (...a: any[]) => any>(f: F) {
  return (...a: Parameters<F>): ReturnType<F> => { console.log(a); return f(...a); };
}
```
**⚖️ Tradeoff** — overloads collapse to the last signature — surprising.
**💥 Prod failure** — wrapper around an overloaded function lost type info on the first overload, callers got `any`.
**🔁 Memory hook** — **"Parameters = tuple of front-door args."**
**🔗 See:** [docs/03](../../docs/03-typescript.md)

### Q13. `Awaited<P>` — why introduced.
**🎯 One-liner** — Recursively unwrap a Promise to its **resolved value type**.
**🧠 Mental model** — Peel the onion of nested Promises.
**📖 Theory**
- Pre-4.5: `Promise<Promise<T>>` would not unwrap fully.
- `Awaited<T>` follows `.then` chains, even nested ones.
- Used internally by `await` itself in newer TS lib.
**💻 Code**
```ts
type A = Awaited<Promise<Promise<number>>>; // number
type B = Awaited<number>;                    // number
```
**⚖️ Tradeoff** — recursion limit can bite very deep generic types.
**💥 Prod failure** — typing `useQuery` result without `Awaited` exposed `Promise<User>` to components → render-as-string.
**🔁 Memory hook** — **"Awaited = peel until done."**
**🔗 See:** [docs/03](../../docs/03-typescript.md)

### Q14. `NonNullable<T>` — implement.
**🎯 One-liner** — Strip `null` and `undefined` from a union.
**🧠 Mental model** — Sieve out the nullish noise.
**📖 Theory**
- Built from `T extends null | undefined ? never : T`.
- Often paired with type guards after a runtime null-check.
**💻 Code**
```ts
type MyNonNullable<T> = T extends null | undefined ? never : T;
type S = MyNonNullable<string | null | undefined>; // string

function assert<T>(x: T): NonNullable<T> {
  if (x == null) throw new Error('null');
  return x as NonNullable<T>;
}
```
**⚖️ Tradeoff** — runtime guard still required; type narrowing alone doesn't protect prod.
**💥 Prod failure** — `NonNullable` cast hid an actually-null API response → `Cannot read property` on first render.
**🔁 Memory hook** — **"NonNullable = no null, no undef."**
**🔗 See:** [docs/03](../../docs/03-typescript.md)

### Q15. Conditional types — `T extends U ? X : Y`.
**🎯 One-liner** — `if/else` for types.
**🧠 Mental model** — A type-level ternary that branches on assignability.
**📖 Theory**
- `T extends U ? X : Y` evaluates to `X` if `T` is assignable to `U`, else `Y`.
- Distributes over unions when `T` is a naked type parameter.
- Combine with `infer` to extract subtypes.
**💻 Code**
```ts
type IsString<T> = T extends string ? true : false;
type A = IsString<'hi'>;   // true
type B = IsString<number>; // false
```
**⚖️ Tradeoff** — distributive behavior surprises beginners (`IsString<string | number>` = `true | false`).
**💥 Prod failure** — generic helper expected a single boolean, got `boolean` (= `true | false`) and broke a switch.
**🔁 Memory hook** — **"types think in if/else too."**
**🔗 See:** [docs/03 §conditional](../../docs/03-typescript.md)

### Q16. Distributive conditional types — gotcha with unions.
**🎯 One-liner** — Conditionals **distribute** over union members; wrap in `[ ]` to stop.
**🧠 Mental model** — Each union arm is processed separately, then re-unioned.
**📖 Theory**
- `T extends U ? X : Y` with `T = A | B` becomes `(A extends U ? X : Y) | (B extends U ? X : Y)`.
- To prevent: `[T] extends [U] ? X : Y`.
**💻 Code**
```ts
type ToArray<T> = T extends any ? T[] : never;
type A = ToArray<string | number>;       // string[] | number[]  (distributed)

type ToArrayNoDist<T> = [T] extends [any] ? T[] : never;
type B = ToArrayNoDist<string | number>; // (string | number)[]  (single)
```
**⚖️ Tradeoff** — distribution is usually what you want for `Exclude`-style helpers.
**💥 Prod failure** — type guard returned `string[] | number[]` where a `(string|number)[]` was expected → consumers couldn't `.push` mixed.
**🔁 Memory hook** — **"naked T distributes, `[T]` stays whole."**
**🔗 See:** [docs/03](../../docs/03-typescript.md)

### Q17. Mapped types — write `DeepReadonly<T>`.
**🎯 One-liner** — Walk every key, transform its modifier or value.
**🧠 Mental model** — A `for-each` loop over a type's keys.
**📖 Theory**
- `{ [K in keyof T]: ... }` is the syntax.
- Add modifiers: `readonly`, `?:`. Remove with `-readonly`, `-?`.
- Recurse into object values for "deep" variants.
**💻 Code**
```ts
type DeepReadonly<T> = {
  readonly [K in keyof T]: T[K] extends object ? DeepReadonly<T[K]> : T[K];
};
const cfg: DeepReadonly<{ a: { b: number } }> = { a: { b: 1 } };
// cfg.a.b = 9; // ❌ readonly
```
**⚖️ Tradeoff** — `Function`/`Date` are objects — recursion may over-apply.
**💥 Prod failure** — `DeepReadonly` froze a function, callers got "not callable" diagnostics.
**🔁 Memory hook** — **"map = for-each-key on a type."**
**🔗 See:** [docs/03 §mapped](../../docs/03-typescript.md)

### Q18. Template literal types — `` `on${Capitalize<E>}` `` example.
**🎯 One-liner** — String literals with **type-level interpolation**.
**🧠 Mental model** — Backticks at the type layer, like JS template strings.
**📖 Theory**
- Combine literals with `${}` placeholders.
- Built-ins: `Uppercase`, `Lowercase`, `Capitalize`, `Uncapitalize`.
- Powerful for typed events, route generators, css-in-ts.
**💻 Code**
```ts
type Event = 'click' | 'hover';
type Handler = `on${Capitalize<Event>}`; // 'onClick' | 'onHover'

type Props = { [K in Handler]?: () => void };
const p: Props = { onClick: () => {}, onHover: () => {} };
```
**⚖️ Tradeoff** — recursion depth limited; very large unions explode compile time.
**💥 Prod failure** — auto-generated route types from a 1k-route map crashed `tsc` with "type instantiation excessively deep."
**🔁 Memory hook** — **"backticks in types = strings + math."**
**🔗 See:** [docs/03](../../docs/03-typescript.md)

### Q19. Discriminated union — write `Result<T>` and narrow it.
**🎯 One-liner** — A union where each arm shares a **literal tag** that lets TS narrow.
**🧠 Mental model** — Each variant carries a flag the type checker reads first.
**📖 Theory**
- Each member has the same property with a different literal value.
- `if (r.kind === 'ok')` narrows to that arm — TS knows `value` is present.
- The cleanest model for "result, error, loading" states.
**💻 Code**
```ts
type Result<T> = { kind: 'ok'; value: T } | { kind: 'err'; error: Error };

function show<T>(r: Result<T>) {
  if (r.kind === 'ok') return r.value;   // narrowed
  return r.error.message;
}
```
**⚖️ Tradeoff** — adding a new arm requires updating every consumer; that's the point.
**💥 Prod failure** — using `success?: T; error?: Error` instead of a tagged union → both true → ambiguous render.
**🔁 Memory hook** — **"one tag, many arms."**
**🔗 See:** [docs/03 §unions](../../docs/03-typescript.md)

### Q20. Exhaustive switch with `never`.
**🎯 One-liner** — Force compile-time error when a new union arm is added.
**🧠 Mental model** — `never` is the "this can't happen" type — assigning to it is the trap.
**📖 Theory**
- After all `case`s are handled, `default` should narrow to `never`.
- Assign the variable to a `never`-typed function param to enforce.
- Catches "I forgot to handle the new variant."
**💻 Code**
```ts
type Shape = { k: 'circle' } | { k: 'square' };
function area(s: Shape): number {
  switch (s.k) {
    case 'circle': return 1;
    case 'square': return 2;
    default: const _e: never = s; return _e; // compile error if Shape grows
  }
}
```
**⚖️ Tradeoff** — only works on tagged unions; arbitrary booleans don't narrow.
**💥 Prod failure** — added a new payment status, missed updating one switch — caught only via Sentry NaN.
**🔁 Memory hook** — **"default → never = exhaustive proof."**
**🔗 See:** [docs/03](../../docs/03-typescript.md)

### Q21. Type guard — `function isUser(x): x is User`.
**🎯 One-liner** — A function whose return type **narrows** the argument in callers.
**🧠 Mental model** — A bouncer who tells the room "this is definitely a User."
**📖 Theory**
- Return type `x is T` is a **type predicate**.
- Inside the guarded `if`, TS narrows.
- Use `in` operator for cheap structural checks; full guards for runtime safety.
**💻 Code**
```ts
function isUser(x: unknown): x is { id: string; name: string } {
  return !!x && typeof (x as any).id === 'string' && typeof (x as any).name === 'string';
}
function greet(x: unknown) { if (isUser(x)) console.log(x.name); }
```
**⚖️ Tradeoff** — guard is only as honest as its body; lying with `as` causes runtime crashes.
**💥 Prod failure** — guard checked `x.id` truthy; empty string passed; downstream `id.split` blew up.
**🔁 Memory hook** — **"`x is T` = compiler-trusted narrowing."**
**🔗 See:** [docs/03 §guards](../../docs/03-typescript.md)

### Q22. `unknown` vs `any` vs `never`.
**🎯 One-liner** — `any` opts out, `unknown` requires proof, `never` says impossible.
**🧠 Mental model** — `any` = no seatbelt · `unknown` = locked box · `never` = empty room.
**📖 Theory**
- `any` disables checking — assignable to/from anything.
- `unknown` is the type-safe `any`: must narrow before use.
- `never` is the bottom type — no value inhabits it; result of exhaustive checks, infinite loops, throws.
**💻 Code**
```ts
let a: any = 1; a.toUpperCase();        // compiles, may crash
let u: unknown = 1; // u.toUpperCase(); // ❌ until narrowed
if (typeof u === 'string') u.toUpperCase();
function bomb(): never { throw new Error(); }
```
**⚖️ Tradeoff** — `any` is contagious; one leak ruins downstream types.
**💥 Prod failure** — `JSON.parse` → `any` → consumed without validation → wrong shape crashed UI.
**🔁 Memory hook** — **"any = open door, unknown = locked door, never = no door."**
**🔗 See:** [docs/03](../../docs/03-typescript.md)

### Q23. `as const` — what it does to `['a','b']`.

### Q24. `satisfies` operator — when it beats `as`.
**🎯 One-liner** — Validate a value matches a type **without widening it**.
**🧠 Mental model** — `as` is a cast (lie if needed); `satisfies` is a proof (must be true).
**📖 Theory**
- `expr satisfies T` checks expr matches T but keeps the **narrower inferred type**.
- `expr as T` overrides — can hide errors.
- Great for config objects + `Record` validation where you also want literal types.
**💻 Code**
```ts
const colors = { primary: '#000', accent: '#fff' } satisfies Record<string, string>;
colors.primary; // type: '#000' (literal preserved)

const colorsAs = { primary: '#000' } as Record<string, string>;
colorsAs.primary; // type: string (widened)
```
**⚖️ Tradeoff** — TS 4.9+ only; older projects can't use it.
**💥 Prod failure** — `as Record<string, Handler>` masked a missing key; `satisfies` would have caught it.
**🔁 Memory hook** — **"as lies, satisfies proves."**
**🔗 See:** [docs/03](../../docs/03-typescript.md)

### Q25. Index signatures vs `Record`.
**🎯 One-liner** — Index signature = "any key allowed"; `Record` can use a fixed key set.
**🧠 Mental model** — Open shelving vs labeled drawers.
**📖 Theory**
