### Q1. Difference between `var`, `let`, `const` in scope, hoisting, TDZ.
**A.**
- `var` → **function-scoped**, hoisted + initialized to `undefined`. No TDZ.
- `let` / `const` → **block-scoped**, hoisted but **not initialized** → access before declaration throws (TDZ).
- `const` → binding is immutable; the value is not (objects still mutable).

```js
console.log(a); // undefined
var a = 1;

console.log(b); // ReferenceError (TDZ)
let b = 2;

const obj = { x: 1 };
obj.x = 2; // OK — only the binding is frozen
```
**Tradeoff:** `var` survives loops (closure-in-loop bug). Default to `const`, fall back to `let`.
**Prod failure:** classic `for (var i…) setTimeout(() => log(i))` prints the final `i` N times.
**See:** [docs/02 §scope](../../docs/02-javascript-core.md)

### Q2. What is the temporal dead zone — give an example that throws.
**A.**
- Window between scope entry and the `let`/`const` declaration where the binding exists but is uninitialized.
- Any read or write throws `ReferenceError`.
- `typeof` on a TDZ binding **also throws** (unlike on undeclared vars).

```js
{
  // TDZ for `x` starts here
  // console.log(x);  // ReferenceError
  // typeof x;        // ReferenceError (!)
  let x = 1;          // TDZ ends
}
```
**Tradeoff:** TDZ catches order-of-init bugs at runtime; downside is `typeof` no longer "safe".
**Prod failure:** circular module imports landing in the middle of TDZ → cryptic "Cannot access 'X' before initialization".
**See:** [docs/02 §scope/TDZ](../../docs/02-javascript-core.md)

### Q3. Primitive vs reference types — list all 7 primitives.
**A.**
- 7 primitives: `string`, `number`, `bigint`, `boolean`, `undefined`, `symbol`, `null`.
- Primitives are **immutable + compared by value**.
- Everything else (objects, arrays, functions, dates, Maps…) is a **reference** — compared by identity.

```js
1 === 1;            // true (value)
[] === [];          // false (different refs)
const a = { n: 1 }; const b = a;
b.n = 2; a.n;       // 2 (same ref)
```
**Tradeoff:** primitive autoboxing (`"a".length`) is cheap but creates wrapper objects in some engines.
**Prod failure:** using an object as a `useState` initial without memoization → new ref each render → infinite loop in `useEffect([obj])`.
**See:** [docs/02 §types](../../docs/02-javascript-core.md)

### Q4. `==` vs `===` — name one case where `==` is "safe".
**A.**
- `===` strict: same type + same value, no coercion.
- `==` loose: applies `ToPrimitive` / `ToNumber` rules — full of footguns.
- The one defensible use: `x == null` to test for `null` **or** `undefined` in a single check.

```js
0 == false;     // true
'' == false;    // true
null == undef;  // true
NaN == NaN;     // false (use Number.isNaN)
x == null;      // ✅ idiomatic null/undefined check
```
**Tradeoff:** even the `== null` shortcut trips most lint configs (`eqeqeq`).
**Prod failure:** `if (count == 0)` matched `""`, `false`, `null` from a malformed API.
**See:** [docs/02 §coercion](../../docs/02-javascript-core.md)

### Q5. Explain `this` in 5 contexts.
**A.**
- **Default**: `undefined` in strict mode, global object in sloppy.
- **Implicit**: `obj.fn()` → `this = obj`.
- **Explicit**: `fn.call(x)` / `apply` / `bind` → `this = x`.
- **`new`**: `new Ctor()` → `this` = newly created object.
- **Arrow**: lexical — captured from enclosing scope, can't be rebound.

```js
const o = { x: 1, get() { return this.x; } };
o.get();              // 1 (implicit)
const g = o.get;
g();                  // undefined (default)
g.call({ x: 9 });     // 9 (explicit)
const arrow = () => this;  // captures outer `this`
```
**Tradeoff:** arrow functions can't be methods if you need dynamic `this`.
**Prod failure:** passing `this.handler` to a callback without `.bind(this)` → `Cannot read property 'state' of undefined`.
**See:** [docs/02 §this](../../docs/02-javascript-core.md)

### Q6. Why arrow functions can't be used as constructors.
**A.**
- Arrows have no `[[Construct]]` internal slot.
- They have no own `this`, no `prototype` property, no `arguments`, no `new.target`.
- `new (() => {})()` throws `TypeError: not a constructor`.

```js
const F = () => {};
new F(); // TypeError
F.prototype; // undefined
```
**Tradeoff:** arrows are great for callbacks, terrible for class-like factories.
**Prod failure:** auto-converting `function` → arrow with a codemod broke a Mongoose schema constructor.
**See:** [docs/02 §functions](../../docs/02-javascript-core.md)

### Q7. Closure — define + write a counter using one.
**A.**
- A function plus the lexical environment captured at its creation.
- Lets the inner function access outer variables after the outer returns.
- Foundation of modules, currying, hooks' state.

```js
function makeCounter() {
  let n = 0;
  return {
    inc: () => ++n,
    get: () => n,
  };
}
const c = makeCounter();
c.inc(); c.inc(); c.get(); // 2
```
**Tradeoff:** closures keep references alive → memory leak if you stash long-lived closures over big objects.
**Prod failure:** event listener closure pinned a 10MB image blob → OOM after 200 navigations.
**See:** [docs/02 §closures](../../docs/02-javascript-core.md)

### Q8. Stale closure inside `setInterval` — show bug + fix.
**A.**
- Interval callback captures the **first** value of `count` and never re-reads.
- Fix in vanilla JS: read from a ref/object. In React: use a ref or functional setState.

```js
// Bug
function start() {
  let count = 0;
  setInterval(() => console.log(count), 500); // always 0
  setTimeout(() => count++, 1000);
}

// Fix (functional update in React)
useEffect(() => {
  const id = setInterval(() => setCount(c => c + 1), 500);
  return () => clearInterval(id);
}, []);
```
**Tradeoff:** using a ref hides re-renders; functional setState is cleaner.
**Prod failure:** debounced search hook called with a stale `query` shipped wrong results to analytics.
**See:** [docs/02 §closures](../../docs/02-javascript-core.md), [docs/04 hooks pitfalls](../../docs/04-react-deep-dive.md)

### Q9. IIFE — what is it, when used today.
**A.**
- *Immediately Invoked Function Expression* — `(function(){…})()`.
- Pre-ES6: created a private scope to avoid polluting globals.
- Today: rare in app code (modules do this); still useful for top-level `await` polyfills, isolating script tags, or `(async () => {…})()` in Node CLIs.

```js
(async () => {
  const data = await fetch('/me').then(r => r.json());
  console.log(data);
})();
```
**Tradeoff:** modern code uses ES modules; IIFEs hurt readability in module files.
**Prod failure:** wrapping a whole React component in an IIFE accidentally executed it at import time.
**See:** [docs/02 §functions](../../docs/02-javascript-core.md)

### Q10. Event loop: stack vs microtask vs macrotask order.
**A.**
- One **call stack**, one **microtask queue** (Promises, `queueMicrotask`, MutationObserver), many **macrotask queues** (`setTimeout`, I/O, `setImmediate`).
- Per tick: pop next macrotask → run to completion → drain **all** microtasks → render → repeat.
- Microtasks always beat the next macrotask.

```js
console.log('A');
setTimeout(() => console.log('B'), 0);
Promise.resolve().then(() => console.log('C'));
console.log('D');
// Output: A D C B
```
**Tradeoff:** microtask floods (recursive `.then`) starve rendering.
**Prod failure:** `await` chain in a render loop blocked frame paint and dropped FPS to 12.
**See:** [docs/02 §event-loop](../../docs/02-javascript-core.md)

### Q11. Output: `console.log(1); setTimeout(..,0); Promise.resolve().then(..); console.log(4)`.
**A.**
- Synchronous lines first → `1`, `4`.
- Microtask before next macrotask → Promise's `.then`.
- Then the timer.
- Order: **1, 4, (Promise), (Timeout)**.

```js
console.log(1);
setTimeout(() => console.log('T'), 0);
Promise.resolve().then(() => console.log('P'));
console.log(4);
// 1 4 P T
```
**Tradeoff:** Hermes implements the same spec but timer floor is ~1ms (not 0).
**Prod failure:** test expecting `setTimeout` to fire before a `Promise.resolve().then` (it never does).
**See:** [docs/02 §event-loop](../../docs/02-javascript-core.md)

### Q12. `Promise.all` vs `allSettled` vs `race` vs `any`.
**A.**
- **`all`** — resolves with array when all resolve; rejects on **first rejection**.
- **`allSettled`** — never rejects; returns `{status, value|reason}[]`.
- **`race`** — first to settle (resolve **or** reject) wins.
- **`any`** — first to **resolve** wins; rejects only if all fail (`AggregateError`).

```js
const r = await Promise.allSettled([fetchA(), fetchB(), fetchC()]);
const ok = r.filter(x => x.status === 'fulfilled').map(x => x.value);
```
**Tradeoff:** `all` cancels nothing — losers keep running. Use `AbortController`.
**Prod failure:** dashboard used `Promise.all`; one flaky widget threw and blanked the whole screen — should've been `allSettled`.
**See:** [docs/02 §async](../../docs/02-javascript-core.md)

### Q13. Implement `Promise.all` from scratch.
**A.**
- Track count of resolved.
- Resolve final promise once `count === inputs.length`.
- Reject immediately on any rejection.

```ts
function pAll<T>(promises: Promise<T>[]): Promise<T[]> {
  return new Promise((resolve, reject) => {
    const out: T[] = new Array(promises.length);
    let done = 0;
    if (promises.length === 0) return resolve(out);
    promises.forEach((p, i) =>
      Promise.resolve(p).then(
        v => { out[i] = v; if (++done === promises.length) resolve(out); },
        reject,
      ),
    );
  });
}
```
**Tradeoff:** doesn't handle iterables; spec version does.
**Prod failure:** custom impl forgot the empty-array case → never resolved.
**See:** [docs/02 §async](../../docs/02-javascript-core.md)

### Q14. Implement `pLimit(n)` — concurrency-limited fetch.
**A.**
- Maintain an in-flight count and a queue of waiting tasks.
- On task finish, dequeue and start the next.
- Returns a wrapper that returns the task's promise.

```ts
function pLimit(n: number) {
  let active = 0;
  const q: Array<() => void> = [];
  const next = () => {
    if (active >= n || q.length === 0) return;
    active++;
    q.shift()!();
  };
  return <T>(fn: () => Promise<T>) =>
    new Promise<T>((res, rej) => {
      q.push(() => fn().then(res, rej).finally(() => { active--; next(); }));
      next();
    });
}
const limit = pLimit(4);
const results = await Promise.all(urls.map(u => limit(() => fetch(u))));
```
**Tradeoff:** simple FIFO; for prioritization use a heap.
**Prod failure:** unbounded `Promise.all(fetch×500)` triggered server rate-limit + RN socket exhaustion.
**See:** [docs/02 §async](../../docs/02-javascript-core.md)

### Q15. Implement `retry(fn, n, delay)` with exponential backoff.
**A.**
- Try once; on failure wait `delay * 2^attempt` (+ jitter).
- Stop after `n` attempts; rethrow last error.
- Always add jitter to avoid thundering herd.

```ts
async function retry<T>(fn: () => Promise<T>, n = 3, base = 300): Promise<T> {
  let last: unknown;
  for (let i = 0; i < n; i++) {
    try { return await fn(); }
    catch (e) {
      last = e;
      const wait = base * 2 ** i + Math.random() * 100;
      await new Promise(r => setTimeout(r, wait));
    }
  }
  throw last;
}
```
**Tradeoff:** retrying non-idempotent POSTs duplicates writes — pair with idempotency key.
**Prod failure:** payment endpoint retried on 500 → double-charge until idempotency added.
**See:** [docs/12 §retries](../../docs/12-networking.md)

### Q16. Implement `debounce` and `throttle`.
**A.**
- **debounce**: delay until input stops for `ms`.
- **throttle**: at most one call per `ms`.

```ts
const debounce = <A extends any[]>(fn: (...a: A) => void, ms: number) => {
  let t: any;
  return (...a: A) => { clearTimeout(t); t = setTimeout(() => fn(...a), ms); };
};

const throttle = <A extends any[]>(fn: (...a: A) => void, ms: number) => {
  let last = 0;
  return (...a: A) => {
    const now = Date.now();
    if (now - last >= ms) { last = now; fn(...a); }
  };
};
```
**Tradeoff:** debounce hides every keystroke; throttle drops them. Pick by UX intent.
**Prod failure:** scroll handler not throttled drove JS thread to 100% on Android low-end.
**See:** [docs/07 §perf](../../docs/07-performance.md)

### Q17. Difference between debounce and throttle — RN use cases.
**A.**
- **Debounce** = "after silence" → search bar autocomplete, form validation.
- **Throttle** = "at most every X" → scroll handler, gesture progress, analytics ping.
- Debounce risks losing trailing call if not configured; throttle risks bursty UX.

```ts
<TextInput onChangeText={debounce(query, 300)} />
<ScrollView onScroll={throttle(track, 200)} />
```
**Tradeoff:** for animations, neither — use Reanimated worklets on UI thread.
**Prod failure:** debounced "Save" with 1s window let user navigate away before save fired.
**See:** [docs/07 §perf](../../docs/07-performance.md)

### Q18. Implement `memoize(fn)`.
**A.**
- Cache by serialized args.
- Return cached result on hit.
- For multi-arg or object args, use `Map` keyed by arg tuple identity, or a stable hash.

```ts
function memoize<A extends any[], R>(fn: (...a: A) => R) {
  const cache = new Map<string, R>();
  return (...a: A): R => {
    const k = JSON.stringify(a);
    if (cache.has(k)) return cache.get(k)!;
    const v = fn(...a);
    cache.set(k, v);
    return v;
  };
}
```
**Tradeoff:** unbounded cache → leak. Use LRU or WeakMap for object args.
**Prod failure:** memoizing on `JSON.stringify(largeObj)` cost more CPU than the call itself.
**See:** [docs/02 §functional](../../docs/02-javascript-core.md)

### Q19. Implement deep clone (handle cycles).
**A.**
- Walk recursively; for objects use a `WeakMap<src, copy>` to detect cycles.
- Handle special types: `Date`, `RegExp`, `Map`, `Set`.
- Modern shortcut: `structuredClone()` (works in Hermes ≥ 0.12.x via polyfill).

```ts
function clone<T>(x: T, seen = new WeakMap()): T {
  if (x === null || typeof x !== 'object') return x;
  if (seen.has(x as any)) return seen.get(x as any);
  const out: any = Array.isArray(x) ? [] : {};
  seen.set(x as any, out);
  for (const k of Object.keys(x as any)) out[k] = clone((x as any)[k], seen);
  return out;
}
```
**Tradeoff:** `JSON.parse(JSON.stringify(x))` drops functions/Dates/`undefined`/cycles.
**Prod failure:** Redux deep-clone via JSON corrupted `Date` fields silently.
**See:** [docs/02 §objects](../../docs/02-javascript-core.md)

### Q20. Implement `curry(fn)`.
**A.**
- Return a function that collects args until arity met, then invokes.

```ts
const curry = (fn: Function) => {
  return function curried(...args: any[]): any {
    return args.length >= fn.length
      ? fn(...args)
      : (...more: any[]) => curried(...args, ...more);
  };
};
const add = curry((a: number, b: number, c: number) => a + b + c);
add(1)(2)(3); // 6
add(1, 2)(3); // 6
```
**Tradeoff:** harder to type in TS; readability cost in app code.
**Prod failure:** curried selector swallowed an extra arg silently → returned partial fn instead of value.
**See:** [docs/02 §functional](../../docs/02-javascript-core.md)

### Q21. Implement `compose` and `pipe`.
**A.**
- **`compose`**: right-to-left function composition `f(g(h(x)))`.
- **`pipe`**: left-to-right `h → g → f`.

```ts
const pipe = (...fns: Function[]) => (x: unknown) => fns.reduce((v, f) => f(v), x);
const compose = (...fns: Function[]) => (x: unknown) => fns.reduceRight((v, f) => f(v), x);
const upper = (s: string) => s.toUpperCase();
const excite = (s: string) => s + '!';
pipe(upper, excite)('hi'); // "HI!"
```
**Tradeoff:** tracing errors through a 7-step pipe is painful.
**Prod failure:** `pipe` mutated input via a non-pure step → silent data corruption.
**See:** [docs/02 §functional](../../docs/02-javascript-core.md)

### Q23. Implement `Array.prototype.reduce` polyfill.
**A.**
- If no initial value, use index 0 as accumulator and start at 1.
- Throw `TypeError` on empty array with no initial value.

```js
Array.prototype.myReduce = function(cb, init) {
  let acc, i = 0;
  if (arguments.length >= 2) acc = init;
  else if (this.length === 0) throw new TypeError('Reduce of empty array with no initial value');
  else { acc = this[0]; i = 1; }
  for (; i < this.length; i++) if (i in this) acc = cb(acc, this[i], i, this);
  return acc;
};
```
**Tradeoff:** misuse of reduce as imperative loop hurts readability.
**Prod failure:** `reduce` without initial accumulator on filtered empty list crashed checkout.
**See:** [docs/02 §arrays](../../docs/02-javascript-core.md)

### Q24. Implement `Function.prototype.bind` polyfill.
**A.**
- Capture `this` + preset args; return a new function.
- Preserve `new`-ability (when called with `new`, ignore bound `this`).

```js
Function.prototype.myBind = function(ctx, ...preset) {
  const fn = this;
  function bound(...args) {
    const isNew = this instanceof bound;
    return fn.apply(isNew ? this : ctx, [...preset, ...args]);
  }
  bound.prototype = Object.create(fn.prototype || null);
  return bound;
};
```
**Tradeoff:** prototype chain plumbing is easy to forget; spec compliance is subtle.
**Prod failure:** custom bind broke `instanceof` on a class subclass.
**See:** [docs/02 §functions](../../docs/02-javascript-core.md)

### Q25. Prototype chain — explain with `Object.create`.
**A.**
- Every object has an internal `[[Prototype]]` link, accessed via `Object.getPrototypeOf`.
- Property lookup walks the chain until found or `null`.
- `Object.create(proto)` makes a new object whose `[[Prototype]]` is `proto`.

```js
const animal = { eats: true };
const dog = Object.create(animal);
dog.barks = true;
dog.eats;            // true (inherited)
Object.getPrototypeOf(dog) === animal; // true
```
**Tradeoff:** deep chains hurt lookup perf and clarity.
**Prod failure:** library extended `Object.prototype` → broke `for...in` everywhere.
**See:** [docs/02 §prototypes](../../docs/02-javascript-core.md)

### Q26. `class` vs prototypal inheritance — what `class` desugars to.
**A.**
- `class` is sugar over constructor function + prototype methods.
- `extends` wires `Child.prototype.__proto__ = Parent.prototype` and `Child.__proto__ = Parent` (static chain).
- `super(...)` calls parent constructor with proper `new.target`.

```js
class A { hi() { return 'A'; } }
class B extends A { hi() { return 'B/' + super.hi(); } }
new B().hi(); // "B/A"
```
**Tradeoff:** `class` syntax can mislead OOP-trained devs into deep hierarchies; composition usually wins.
**Prod failure:** transpiled class without `super()` in subclass crashed in Hermes only.
**See:** [docs/02 §classes](../../docs/02-javascript-core.md)

### Q27. `Object.freeze` vs `Object.seal` vs `const`.
**A.**
- `const` → can't **reassign** the binding; value still mutable.
- `Object.seal(o)` → can't add/remove props; existing props writable.
- `Object.freeze(o)` → can't add/remove/modify (shallow only).

```js
const o = Object.freeze({ a: 1, nested: { b: 2 } });
o.a = 9;          // ignored (strict: throws)
o.nested.b = 9;   // ✅ mutates (shallow!)
```
**Tradeoff:** `freeze` is shallow; deep-freeze costs memory + walk time.
**Prod failure:** Redux assumed frozen state but only top-level was frozen → mutation slipped in.
**See:** [docs/02 §objects](../../docs/02-javascript-core.md)

### Q28. Shallow vs deep equality — write `deepEqual`.
**A.**
- Shallow: compare top-level keys with `===`.
- Deep: recurse into nested objects/arrays; handle special types and cycles.

```ts
function deepEqual(a: any, b: any, seen = new WeakMap()): boolean {
  if (Object.is(a, b)) return true;
  if (typeof a !== 'object' || typeof b !== 'object' || a === null || b === null) return false;
  if (seen.get(a) === b) return true;
  seen.set(a, b);
  const ka = Object.keys(a), kb = Object.keys(b);
  if (ka.length !== kb.length) return false;
  return ka.every(k => deepEqual(a[k], b[k], seen));
}
```
**Tradeoff:** O(n) per compare — avoid in render hot paths.
**Prod failure:** `useEffect` dep array compared deep objects every render → 90% of frame in equality.
**See:** [docs/04 §memo](../../docs/04-react-deep-dive.md)

### Q29. `Map` vs `Object` — when each.
**A.**
- **`Map`**: any-type keys, insertion order, true `size`, optimized for frequent add/delete.
- **`Object`**: string/symbol keys, prototype pollution risk, JSON-friendly.
- Use `Map` for caches/lookups; `Object` for record-shaped data.

```ts
const m = new Map<string, User>();
m.set('u1', user); m.get('u1');
```
**Tradeoff:** `Map` doesn't serialize via `JSON.stringify` directly.
**Prod failure:** using object as cache, key `"toString"` collided with prototype method.
**See:** [docs/02 §collections](../../docs/02-javascript-core.md)

### Q30. `Set` vs array — perf characteristics.
**A.**
- `Set` → O(1) add/has/delete; preserves insertion order; unique values.
- `Array` → O(n) `includes`, but cache-friendly + index access.
- Convert with `[...set]` / `new Set(arr)`.

```ts
const seen = new Set<string>();
ids.filter(id => !seen.has(id) && seen.add(id));
```
**Tradeoff:** small N (< ~30) — array is faster due to cache + JIT.
**Prod failure:** O(n²) dedupe via `arr.filter(includes)` melted CPU on a 50k list.
**See:** [docs/02 §collections](../../docs/02-javascript-core.md)

### Q31. WeakMap / WeakSet — use case.
**A.**
- Keys must be objects; **weakly held** → don't prevent GC.
- Not iterable, no `size`.
- Use for "tag this object without modifying it" or memoize-by-object.

```ts
const meta = new WeakMap<object, Metadata>();
meta.set(component, { mountedAt: Date.now() });
// no manual cleanup needed when component is GC'd
```
**Tradeoff:** non-deterministic GC timing — can't observe when a key is collected.
**Prod failure:** caching via `Map` (strong) held DOM/native nodes alive after teardown → leak.
**See:** [docs/02 §collections](../../docs/02-javascript-core.md)

### Q32. Symbol — two real uses.
**A.**
- **Unique property keys** → avoid collision in shared objects (e.g., framework metadata).
- **Well-known symbols** → customize built-in behavior (`Symbol.iterator`, `Symbol.toPrimitive`).

```ts
const Brand = Symbol('brand');
class Token { [Brand] = true; }
function isToken(x: any) { return x?.[Brand] === true; }

class Range { *[Symbol.iterator]() { for (let i = 0; i < 3; i++) yield i; } }
[...new Range()]; // [0,1,2]
```
**Tradeoff:** Symbols don't survive `JSON.stringify` and are invisible to `Object.keys`.
**Prod failure:** library used `Symbol.for('redux')` collisions across micro-frontends.
**See:** [docs/02 §symbols](../../docs/02-javascript-core.md)

### Q33. Iterators + generators — `function*` example.
**A.**
- Iterator: object with `next() → {value, done}`.
- Generator function (`function*`) yields values lazily; auto-implements iterator + iterable protocols.
- Powers `for...of`, spread, destructuring on custom data.

```js
function* range(start, end) {
  for (let i = start; i < end; i++) yield i;
}
[...range(1, 4)]; // [1,2,3]
```
**Tradeoff:** generators carry state; not free — avoid in hot loops.
**Prod failure:** infinite generator without `break` froze UI thread.
**See:** [docs/02 §iterators](../../docs/02-javascript-core.md)

---

### Q34. `for...in` vs `for...of` vs `forEach`.
**A.**
- `for...in` → enumerable **keys** (incl. inherited) — almost never what you want for arrays.
- `for...of` → values of any iterable (arrays, strings, Maps, Sets, generators).
- `forEach` → callback per element; can't `break`/`return`.

```js
for (const k in obj) { /* keys */ }
for (const v of arr) { /* values, supports break */ }
arr.forEach((v, i) => { /* no break */ });
```
**Tradeoff:** `forEach` is harder to short-circuit and to `await` correctly.
**Prod failure:** `await` inside `forEach` ran in parallel — order assumption broken.
**See:** [docs/02 §iteration](../../docs/02-javascript-core.md)

---

### Q35. Spread vs rest — show 3 uses each.
**A.**
- **Spread** *expands* an iterable in call/array/object position.
- **Rest** *collects* remaining args/elements into an array/object.

```js
// Spread
Math.max(...nums);
const both = [...a, ...b];
const next = { ...prev, name: 'x' };

// Rest
function sum(...n) { return n.reduce((a,b)=>a+b, 0); }
const [head, ...tail] = arr;
const { id, ...rest } = user;
```
**Tradeoff:** object spread is shallow — nested mutations still shared.
**Prod failure:** `{...state, list: state.list}` shared the inner array → mutation downstream nuked Redux.
**See:** [docs/02 §syntax](../../docs/02-javascript-core.md)

---

### Q36. Destructuring with defaults + renaming.
**A.**
- Defaults apply only when value is `undefined` (not `null`).
- Rename with `:`. Combine: `{ a: localA = 1 }`.

```js
const { a: localA = 1, b = 2 } = { b: null };
// localA = 1, b = null  (default did NOT apply for null)
const [first = 'x', ...rest] = arr;
```
**Tradeoff:** deep destructuring with defaults gets unreadable fast.
**Prod failure:** `{ count = 0 }` didn't default when API sent `null` → arithmetic on `null` → `NaN`.
**See:** [docs/02 §syntax](../../docs/02-javascript-core.md)

### Q37. Optional chaining + nullish coalescing — `??` vs `||` trap.
**A.**
- `?.` short-circuits to `undefined` on `null`/`undefined`.
- `??` falls back **only** for `null`/`undefined`.
- `||` falls back for any falsy (`0`, `''`, `false`).

```js
const port = settings?.port ?? 3000;   // 0 stays 0
const portBuggy = settings?.port || 3000; // 0 → 3000 (!)
```
**Tradeoff:** `??` does not work mixed with `||`/`&&` without parens.
**Prod failure:** `||` on `count` defaulted `0` to `10` → wrong charge amount.
**See:** [docs/02 §syntax](../../docs/02-javascript-core.md)

### Q38. JSON.stringify gotchas (functions, undefined, BigInt, circular).
**A.**
- Functions, `undefined`, Symbols → **omitted** in objects, **null** in arrays.
- `BigInt` → throws `TypeError`.
- Circular refs → throws.
- `Date` → ISO string; loses type info.

```js
JSON.stringify({ a: undefined, b: 1n }); // throws on bigint
JSON.stringify([undefined]);             // "[null]"
const o = {}; o.self = o;
JSON.stringify(o);                       // throws (circular)
```
**Tradeoff:** custom `replacer` mitigates but is easy to misuse.
**Prod failure:** logging a circular response object crashed Sentry beforeSend.
**See:** [docs/02 §json](../../docs/02-javascript-core.md)

### Q39. Memory leak sources in JS (timers, listeners, closures, detached views).
**A.**
- Uncleaned timers / intervals.
- Event listeners / subscriptions not removed.
- Long-lived closures over big objects.
- Detached native views/DOM still referenced from JS.
- Global caches without eviction.

```ts
useEffect(() => {
  const sub = emitter.addListener('x', handler);
  const id = setInterval(tick, 1000);
  return () => { sub.remove(); clearInterval(id); };
}, []);
```
**Tradeoff:** "fix on unmount" pattern is verbose but mandatory.
**Prod failure:** RN screen leaked 8MB per nav due to `KeyboardEvent` listener not removed.
**See:** [docs/09 §leaks](../../docs/09-debugging.md)

### Q40. Garbage collection — mark and sweep, generational hints.
**A.**
- Roots → mark reachable objects → sweep unmarked.
- Generational: most objects die young → "young gen" collected often, cheap; "old gen" rarely.
- Hermes uses a generational GC tuned for mobile.

**Tradeoff:** GC pauses cause jank; allocate less in hot paths.
**Prod failure:** allocating new objects per frame in a Reanimated worklet caused 80ms GC stalls.
**See:** [docs/06 §hermes-gc](../../docs/06-hermes-metro-bundle.md)

### Q41. `async/await` error handling vs `.catch`.
**A.**
- `await` rejection → throws synchronously inside the async fn → catch with `try/catch`.
- `.catch` chain returns a new promise; easier composition.
- Forgetting `await` swallows rejections (becomes unhandled).

```ts
try { const r = await api(); } catch (e) { log(e); }
api().then(use).catch(log);
```
**Tradeoff:** `try/catch` blocks wrap multiple awaits but obscure which one threw.
**Prod failure:** missing `await` on telemetry `flush()` → batch silently dropped on backgrounding.
**See:** [docs/02 §async](../../docs/02-javascript-core.md)

### Q42. Top-level await — works in modules?
**A.**
- Yes, in ES Modules (`type: "module"` / `.mjs`).
- Not in CommonJS, not in script tags without `type="module"`.
- Hermes/Metro: works in app code via the Metro pipeline; behavior depends on bundler config.

```js
// foo.mjs
export const data = await loadConfig();
```
**Tradeoff:** blocks importer evaluation → can deadlock cycles.
**Prod failure:** TLA in shared module delayed RN bundle parse → 600ms cold start regression.
**See:** [docs/02 §modules](../../docs/02-javascript-core.md)

### Q43. ES modules vs CommonJS — interop pitfalls in RN/Metro.
**A.**
- ESM: static `import`/`export`, tree-shakable.
- CJS: dynamic `require`/`module.exports`, default-export interop quirks.
- Metro handles both but **default-import a CJS module** can yield `{ default: actual }` shape.

```js
// CJS dep
module.exports = function fn() {}
// ESM consumer
import fn from 'dep';        // sometimes fn.default needed
import * as ns from 'dep';
```
**Tradeoff:** mixing causes `fn is not a function` only at runtime.
**Prod failure:** a lib shipped CJS only; Metro built fine but RN crashed at first call.
**See:** [docs/06 §metro](../../docs/06-hermes-metro-bundle.md)

### Q44. Tree shaking — what enables it.
**A.**
- Static ESM imports/exports (no `require`).
- `sideEffects: false` in package.json (or per-file glob).
- Pure top-level — no execution side-effects on import.
- Production minifier with DCE.

**Tradeoff:** Metro's tree-shaking is weaker than Webpack/Rollup; rely on inline-requires + Hermes for unused-code elimination.
**Prod failure:** moment.js full-locale import added 200KB.
**See:** [docs/06 §bundle](../../docs/06-hermes-metro-bundle.md)

### Q45. Tagged template literals — example.
**A.**
- A function preceded by a template literal receives `(strings, ...values)`.
- Used by styled-components, GraphQL `gql`, SQL builders.

```ts
function html(strings: TemplateStringsArray, ...vals: unknown[]) {
  return strings.reduce((acc, s, i) => acc + s + (vals[i] != null ? escape(vals[i]) : ''), '');
}
const out = html`<p>Hi ${user.name}</p>`;
```
**Tradeoff:** custom interpolation hides escaping bugs.
**Prod failure:** untagged template injected raw HTML → XSS in WebView.
**See:** [docs/02 §strings](../../docs/02-javascript-core.md)

### Q46. Proxy + Reflect — one real use case.
**A.**
- `Proxy` intercepts ops on a target (`get`, `set`, `has`, …).
- `Reflect` mirrors default behavior — handy inside trap to "do the normal thing".
- Real use: observable stores (Vue 3, Valtio, MobX 6), schema validators, deprecation shims.

```ts
const observe = <T extends object>(t: T, on: (k: string) => void) =>
  new Proxy(t, {
    get(o, k) { on(String(k)); return Reflect.get(o, k); },
  });
```
**Tradeoff:** Hermes Proxy support exists but with perf cost; debuggers struggle.
**Prod failure:** Proxy on a 5k-key object made every render dependency tracker scan O(n).
**See:** [docs/02 §meta](../../docs/02-javascript-core.md)

### Q47. `eval` and `new Function` — security risks.
**A.**
- Both execute strings as code in current/global scope.
- `eval` sees lexical scope (worse).
- Risk: arbitrary code execution if input is attacker-influenced (e.g., remote config).

```js
// Don't:
eval(userInput);
new Function('return ' + userInput)();
```
**Tradeoff:** breaks CSP, blocks minifiers, kills Hermes static optimizations.
**Prod failure:** "dynamic rule" engine eval'd remote strings → RCE via CDN compromise.
**See:** [docs/16 §xss](../../docs/16-mobile-security.md)

### Q48. Strict mode — what it changes.
**A.**
- Implicit `this` in functions is `undefined` (not global).
- Assigning to undeclared vars throws.
- Duplicate parameter names throw.
- `with` is forbidden.
- ES Modules are strict by default.

**Tradeoff:** legacy code may break; default for any modern bundle.
**Prod failure:** silent global creation in non-strict third-party JS clobbered `length`.
**See:** [docs/02 §strict](../../docs/02-javascript-core.md)

### Q49. `typeof null` returns what — why.
**A.**
- Returns `"object"`. Historical bug: in early V1 of JS, type tags were 3 low bits; `null` shared the object tag (0).
- Fixing it would break the web; spec froze the behavior.
- Use `x === null` to check.

```js
typeof null;            // "object"
null instanceof Object; // false
```
**Tradeoff:** linters force explicit `=== null` checks.
**Prod failure:** `typeof user === 'object' && user.name` crashed when `user` was `null`.
**See:** [docs/02 §types](../../docs/02-javascript-core.md)

### Q50. Difference between `null` and `undefined` — when each appears.
**A.**
- **`undefined`**: declared but unassigned, missing function param, missing object prop, function return without value.
- **`null`**: explicit "no value" — set by you/the API.
- `null == undefined` is true; `null === undefined` is false.

```js
let x;            // undefined
const o = {};
o.a;              // undefined
const y = null;   // intentional empty
```
**Tradeoff:** TS `strictNullChecks` separates them; use a project convention (prefer `null` for API, `undefined` for absence).
**Prod failure:** API switched `null` → `undefined` mid-version → forms unfilled because `?? ''` only triggered for one shape.
**See:** [docs/02 §types](../../docs/02-javascript-core.md)
